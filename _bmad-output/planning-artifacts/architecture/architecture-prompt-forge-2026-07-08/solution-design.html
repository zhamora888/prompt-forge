Solution design — for review

How PromptForge is put together, and why

A one-day build, walked through the way the decisions actually got made — including the trade-offs that were considered and rejected, not just the ones that were kept.

§1 Vision

PromptForge is a personal, offline-first prompt library for someone who works across several AI tools in a day. The entire product is one loop: save a prompt once, find it in seconds, copy it, paste it into whatever tool is open. No accounts, no server, no sync — everything lives on one phone.

That constraint — genuinely nothing over the network, ever — is the single biggest lever on everything below. A large share of what a typical mobile architecture has to solve (auth, offline/online reconciliation, API contracts, caching) simply doesn't exist here. What's left is a small, real problem: keep one local list of prompts consistent under a five-screen UI.

§2 Design paradigm

Layered, four layers, one direction of dependency.

Presentationscreens + components
→
StatePromptsProvider
→
RepositorypromptRepository
→
StorageAsyncStorage

No layer skips forward or reaches backward. A screen never touches AsyncStorage or the Repository directly — it only ever calls one hook, usePrompts(). That single rule is what the rest of the spine hangs on: with one owner at every hand-off, there's no seam where two screens built at different times could quietly disagree about how data is read, written, or shaped.

§3 Key decisions

The real forks — shown with what was considered and rejected, not just the conclusion.

Managed Expo chosen

✓
Expo (managed)Native builds, linking, and platform config handled for you. Live testing on-device via Expo Go, zero native build step.
–
Bare React Native CLIFull native control, but requires a working Xcode/Android Studio toolchain and manual linking.

Why: first React Native project, one-day timebox, and — the deciding fact — the build machine isn't set up for native mobile development. Expo Go removes the native toolchain from the critical path entirely.

AsyncStorage, one JSON blob chosen

✓
AsyncStorageSimplest possible API, works in Expo Go with zero setup, no schema. One JSON array holding the whole library, filtered in JS.
–
expo-sqliteA real relational database — more correct for data with relationships, but Prompt has none. Schema/migration overhead this app doesn't need.
–
react-native-mmkv~20–30× faster than AsyncStorage, but a native module — breaks Expo Go, needs a custom dev-client build for a performance win this app's data size will never notice.

Why: a personal library, one flat entity, no relationships, no meaningful scale. "Not designed to scale — just a personal app for showcasing," in the builder's own words. The fastest option was explicitly rejected because it would have cost the one thing that actually matters here: staying inside Expo Go.

Expo Router chosen

✓
Expo RouterFile-based routing, the current default for new Expo apps (SDK 56). Maps almost exactly onto the app's 3-screen, one-level-deep IA.

Why: no real trade-off here — it's the paved path, and nothing about this app's navigation needs anything hand-rolled React Navigation would offer instead.

Single Repository module chosen

✓
One data-access gatewayEvery read/write to AsyncStorage goes through exactly one module. No screen touches storage directly.

Why: "I don't want anyone else messing with data, except that module" — the builder's own framing, and the one architectural rule this spine actually enforces in stone (AD-1, below).

React Context, no state library chosen

✓
React ContextOne provider, one usePrompts() hook, no new dependency.
–
Zustand / ReduxSolves problems — complex cross-cutting state, middleware, devtools — this 3-screen app doesn't have.

Why: three screens, one entity. A state library would be solving a problem that doesn't exist here.

§4 Invariants (AD-1–3)

The rules a future screen can't safely violate, even by accident.

AD-1

Single storage gateway

Prevents
Two screens serializing or mutating the stored JSON differently, corrupting or desyncing the library.
Rule
All AsyncStorage access goes through lib/promptRepository.ts. Every write acts on the single in-memory list PromptsProvider already holds — never a fresh re-read — which is what closes a subtle race: two near-simultaneous actions (say, a delete and a favorite-toggle) apply in sequence against one shared copy of the data, instead of each reading a stale snapshot and silently overwriting the other's change.
AD-2

Single state provider

Prevents
A screen holding its own local copy of the list that drifts from what's actually stored.
Rule
One PromptsProvider, one usePrompts() hook. Search, Favorites-only, and Category filter are three independent fields that AND together — never one exclusive mode — so search and a filter always combine rather than fight each other.
AD-3

Navigation mirrors the UX exactly

Prevents
Navigation drifting from the one-level-deep stack the UX design already committed to.
Rule
Library, Detail, and Edit are sibling routes — Edit is never nested inside Detail's folder. Opening Edit from Detail replaces Detail on the stack rather than pushing on top of it, so the stack is never more than one screen deep from Library.
This exact rule caught a real bug during review — the first draft's file layout violated it. Full story in §7.

  DetailActionRow.tsx  FormInput.tsx

# lib/ — the app's only two "rule enforcers" (AD-1, AD-2), plus shared constants
lib/
  promptRepository.ts       # sole AsyncStorage gateway (AD-1)
  PromptsProvider.tsx        # Context + usePrompts() (AD-2)
  categories.ts               # the fixed 6-category list, single source
  theme.ts                     # design tokens, single source

types/
  prompt.ts                 # the one entity in this app

Prompt is the only entity — id, title, content, category, tags[], isFavorite, createdAt, updatedAt. No relationships to diagram. The Repository is the only code allowed to construct or mutate one.

§7 How this was reviewed

Three independent passes ran against the spine before it was called done.

A rubric pass judged the spine against a good-architecture checklist. A version-verification pass independently re-checked every named technology and version against the live web, rather than trusting what was asserted while drafting. An adversarial pass tried to construct concrete scenarios where two screens, each following every rule to the letter, could still end up building something inconsistent.

What the adversarial pass actually caught

High
The Edit route was nested under Detail's folder in the first draft, directly contradicting the "one level deep" navigation rule stated two paragraphs above it in the same document.
flattened Edit to a sibling route; Edit now replaces Detail on the stack instead of pushing on top
High
No rule specified how concurrent writes resolve — two rapid actions (e.g. delete + favorite-toggle) could each read a stale copy of the list and silently undo each other.
writes now always act on the one in-memory list the app already holds, never a fresh re-read — the race has nowhere to occur
High
The category filter's "off" state and a prompt's "no category" value could have used the same sentinel, making "clear the filter" and "show only uncategorized prompts" ambiguous.
typed as two distinct values — null for "filter off" vs. '' for "prompt has no category"
High
Whether favoriting a prompt should count as "editing" it (and bump the last-modified timestamp) was never decided anywhere.
favoriting is now its own Repository function and explicitly does not touch updatedAt — favoriting isn't editing content
Medium
The single-source-of-truth discipline was applied to data (AD-1) but not to the app's visual tokens — colors and spacing had no equivalent single home in code.
added lib/theme.ts as the one place design tokens live; components reference it rather than hardcoding values
Low
A newly-added dependency (expo-splash-screen) was the only Stack entry without an independent currency check, and turned up an open upstream issue on a preview build.
documented a plain-render fallback in case the splash-screen approach proves unreliable at implementation time

§8 Deferred

Decided not to decide yet — and why that's safe.

AI-assisted prompt improvement.The product vision names it, but it needs an LLM integration this build doesn't have. No AI/network layer exists anywhere in this stack yet — adding one is a future revision, not a gap in this one.
App store submission / production build.Out of scope for a showcase build. Runs via Expo Go / a local dev build.
Dark mode.Deferred at the design layer. Tokens are already structured so adding dark values later doesn't require restructuring anything.
Data export/backup.An accepted risk for v1 — uninstalling the app loses the library. The Repository is the natural extension point if this changes.
Automated testing.Not decided, out of scope for a one-day build. The Repository is the obvious unit-test seam if this gets picked up later.
